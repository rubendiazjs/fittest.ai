# Documentation Roadmap

> Status: Historical. This file captures earlier documentation planning and is kept for context only.

## Why It Is Here

This roadmap predates the current docs rewrite. At the time it was written, the repo still needed accurate top-level docs for setup and architecture.

## What Still Matters

- Keep documentation close to code.
- Update docs in the same change as behavior changes.
- Treat planning docs as planning, not implementation truth.
- Add deeper docs only when the product surface justifies them.

## What Changed Since This Was Written

- `README.md`, `docs/SETUP.md`, `docs/ARCHITECTURE.md`, `docs/SUPABASE.md`, and `docs/TESTING.md` now exist as the current implementation-facing docs.
- Planning material now lives under `docs/planning/`.
- Process guidance now lives under `docs/process/`.
- Templates now live under `docs/templates/`.

## Practical Rule

If you need the current repo truth, use the implementation-facing docs in `docs/` plus the code. If you need historical planning context, use the files in `docs/planning/`.
